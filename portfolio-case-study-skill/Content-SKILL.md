# Portfolio Case Study Builder Skill

## What This Skill Does

This skill helps you create **professional, high-quality case studies** for your portfolio by analyzing excellent examples and providing proven structures, content templates, and writing guidance.

**Use this skill when:**
- Building a new case study for your portfolio
- Improving existing project documentation
- Analyzing successful portfolios for patterns
- Planning the structure of a project presentation
- Writing content for Behance, Dribbble, Medium, or your personal site

---

## How to Use This Skill

### Step 1: Understand the Framework
Read `CASE-STUDY-FRAMEWORK.md` to learn:
- Universal case study structure (extracted from professional portfolios)
- Content principles (specificity, process, visuals)
- Writing style guidelines
- Layout patterns

### Step 2: Choose Your Template
Open `CONTENT-TEMPLATES.md` and select the template that matches your project:
- **Template 1:** UX/Product Design (apps, web apps, UX-heavy projects)
- **Template 2:** Branding/Visual Design (brand identity, marketing sites)
- **Template 3:** Development/Technical (apps, platforms, technical products)
- **Template 4:** Quick Case Study (portfolio grid, shorter projects)

### Step 3: Analyze Reference Examples
Study the analyzed portfolios:
- **Josie Grimmett (UWGLA):** Visual-heavy, brand-focused, story-driven
- **Florinda Sgueglia (Knowlex):** Process-heavy, UX-focused, methodical
- **YouTube Portfolio Patterns:** Navigation, video integration, interaction

### Step 4: Build Your Case Study
Use the templates as a starting point, then:
1. Gather your project materials (visuals, process docs, outcomes)
2. Fill in the template sections with your specific content
3. Follow the content writing tips for each section
4. Apply the visual content checklist
5. Review against quality markers

## Content Quality Checklist

Before publishing your case study, verify:

### Content Completeness
- [ ] Project title and context
- [ ] Your role clearly stated
- [ ] Problem/opportunity defined
- [ ] Process documented with visuals
- [ ] Solution explained with reasoning
- [ ] Outcomes/impact shared
- [ ] Personal reflection included

### Writing Quality
- [ ] Specific, not vague ("reduced by 90%" not "improved a lot")
- [ ] Active voice ("I created" not "was created")
- [ ] Shows thinking ("I chose X because Y")
- [ ] No buzzwords without substance
- [ ] Proper grammar and spelling
- [ ] Readable on mobile devices

### Visual Quality
- [ ] High-resolution images
- [ ] Consistent styling across visuals
- [ ] Before/after comparisons where relevant
- [ ] Process artifacts (flows, wireframes, etc.)
- [ ] Real content (no lorem ipsum)
- [ ] Accessible alt text for images

### Technical Quality
- [ ] All links work
- [ ] Images load quickly
- [ ] Responsive on different screen sizes
- [ ] Navigation is clear
- [ ] Interactive elements function
- [ ] Fast page load

## Common Mistakes to Avoid

### âŒ Mistake 1: Only Showing Final Designs
**Problem:** Readers can't see your thinking or process  
**Fix:** Include research, wireframes, iterations, and rationale

### âŒ Mistake 2: Vague Descriptions
**Problem:** "We improved the user experience"  
**Fix:** "We reduced checkout time from 3 minutes to 45 seconds by removing 2 steps"

### âŒ Mistake 3: Too Much Text
**Problem:** Walls of text that readers skip  
**Fix:** Use visuals to tell the story, text to explain key decisions

### âŒ Mistake 4: No Context
**Problem:** Jumping straight to solutions without explaining the problem  
**Fix:** Always start with WHY this project mattered

### âŒ Mistake 5: Taking All Credit
**Problem:** Claiming sole ownership of team work  
**Fix:** Be clear about your role and give credit to collaborators

### âŒ Mistake 6: Hiding Constraints
**Problem:** Making it seem like everything went perfectly  
**Fix:** Share challenges and how you adapted - it shows problem-solving

### âŒ Mistake 7: Generic Stock Content
**Problem:** Using placeholder images or lorem ipsum  
**Fix:** Only show real project content

### âŒ Mistake 8: No Outcomes
**Problem:** Case study ends after "we launched"  
**Fix:** Share what happened after - feedback, metrics, learnings
